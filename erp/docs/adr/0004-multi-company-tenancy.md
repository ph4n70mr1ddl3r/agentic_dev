# ADR-0004 — Multi-company from day one (legal entities as first-class)

- **Status:** Accepted
- **Date:** 2026-07-12
- **Deciders:** Product Owner, Solution Architect, Data Architect
- **Owner:** Data Architect

## Context

Dynamics 365 (our reference model) is fundamentally **multi-company**: legal
entities have their own ledgers, charts of accounts, fiscal calendars, financial
dimensions, number sequences, and posting profiles. Financials are meaningless
without a company context. Even if v1 ships with a single company, assuming
single-tenancy in the data model would force a high-risk migration later.

## Decision

- **Legal entities are first-class metadata**, not a deployment detail.
- v1 uses a **shared database/schema** with **mandatory `company_id` on every
  transactional table** (`ent_*`), enforced by the data-service layer via
  Postgres **Row-Level Security** and mandatory query scoping.
- Cross-company data (master data like products, shared reference data) is
  explicitly flagged as such in metadata; otherwise data is company-scoped.
- **Database-per-tenant** remains a *future deployment option* and is explicitly
  **not** an API or data-model concern — the service contracts are identical.

## Consequences

**Positive**
- Correct domain model from day one; no painful financial-schema migration.
- Natural fit for financials (per-company ledger/dimensions/posting).
- Inter-company scenarios (e.g. inter-company PO/SO) are representable.

**Negative / costs**
- Every transactional query/path must scope by `company_id` — enforced by
  RLS + tests; discipline required.
- Seed/demo data must always specify a company.

## Alternatives considered

### Alternative A — Single-tenant now, refactor to multi-company later
Rejected: financial schema (ledger, dimensions, postings) is exactly the hardest
thing to migrate; doing it twice is high-risk and expensive.

### Alternative B — Database-per-tenant from day one
Rejected: operational overhead (migrations × N, cross-tenant reporting) before
it's needed; deferred to a deployment decision that doesn't affect contracts.

## Compliance / follow-up

- RLS policies are generated by the data-service for every `ent_*` table.
- A test asserts no query path returns cross-company data unintentionally.
- Master-vs-transactional flag is part of the entity metadata schema.
